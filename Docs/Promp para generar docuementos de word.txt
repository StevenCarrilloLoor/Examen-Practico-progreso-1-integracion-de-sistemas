## 1. HERRAMIENTA DE GENERACIÓN - OBLIGATORIO

### ⚠️ CRÍTICO: SIEMPRE USA JAVASCRIPT CON DOCX-JS

**NUNCA uses:**
- ❌ Python con python-docx
- ❌ Bash scripts para documentos
- ❌ Otras herramientas no validadas

**SIEMPRE usa:**
- ✅ JavaScript con `docx-js` (`npm install -g docx`) para .docx
- ✅ Python con `openpyxl` para .xlsx
- ✅ Python con `python-pptx` para .pptx
- ✅ Python con librerías apropiadas para PDFs

**¿Por qué docx-js?**
- Es la herramienta oficial recomendada en el SKILL.md del entorno
- Tiene validación integrada (`scripts/office/validate.py`)
- Soporte completo para Headings nativos, TOC, tablas, imágenes
- Flujo probado: crear → validar → reparar si falla
- Mejor compatibilidad con Word y Google Docs

### 📦 Setup básico:

```javascript
const { Document, Packer, Paragraph, TextRun, Table, TableRow, TableCell, ImageRun,
        Header, Footer, AlignmentType, PageOrientation, LevelFormat, ExternalHyperlink,
        InternalHyperlink, Bookmark, FootnoteReferenceRun, PositionalTab,
        PositionalTabAlignment, PositionalTabRelativeTo, PositionalTabLeader,
        TabStopType, TabStopPosition, Column, SectionType,
        TableOfContents, HeadingLevel, BorderStyle, WidthType, ShadingType,
        VerticalAlign, PageNumber, PageBreak } = require('docx');
const fs = require('fs');

const doc = new Document({ sections: [{ children: [/* content */] }] });
Packer.toBuffer(doc).then(buffer => fs.writeFileSync("documento.docx", buffer));
```

### 🔍 Validación post-creación (OBLIGATORIA):

```bash
python scripts/office/validate.py documento.docx
```

Si la validación falla, desempaqueta, corrige el XML y reempaqueta:
```bash
python scripts/office/unpack.py documento.docx unpacked/
# corregir XML
python scripts/office/pack.py unpacked/ documento_fixed.docx --original documento.docx
```

## 4. IDENTIFICACIÓN DEL TIPO DE TRABAJO - CRUCIAL

### 🎨 TIPOS DE DOCUMENTOS:

#### A) RESPUESTA ARGUMENTATIVA / PREGUNTA ESPECÍFICA
```
Características:
- Longitud: 2-4 páginas
- Sin estructura formal de ensayo
- Carátula + desarrollo directo
- Responde pregunta específica
- Argumentación enfocada

Estructura:
- Carátula
- Pregunta (centrada y en negrita)
- Desarrollo argumentativo (10-15 párrafos)
- Bibliografía
```

#### B) ENSAYO COMPLETO
```
Características:
- Longitud: 2-5 páginas (según requisitos)
- Estructura formal completa
- Introducción + Desarrollo + Conclusión
- Análisis profundo del tema

Estructura:
- Carátula
- Título del ensayo
- Introducción (contexto, planteamiento)
- Desarrollo (2-3 secciones con subtítulos)
- Conclusión (síntesis y reflexión)
- Bibliografía
```

#### C) TAREA EXTRACURRICULAR / CUESTIONARIO
```
Características:
- Múltiples secciones/actividades
- Preguntas conceptuales + casos prácticos
- Formato: pregunta → respuesta

Estructura:
- Carátula
- Actividad 1 con preguntas numeradas
- Actividad 2 con casos numerados
- Actividad 3 con desarrollo
- Bibliografía
```

#### D) DOCUMENTOS ESPECIALES
```
- Mapas conceptuales: Usar tablas, diagramas, relaciones visuales
- Cuadros comparativos: Tablas con columnas comparativas
- Informes técnicos: Secciones, subsecciones, datos, gráficos
- Análisis de casos: Hechos → Problema → Análisis → Solución
- Guías/Manuales: Pasos numerados, instrucciones claras
```

### 🎯 REGLA DE ORO:
**Lee las instrucciones cuidadosamente y determina QUÉ tipo de documento necesita. NO asumas que todo es un ensayo.**

## 6. HEADINGS NATIVOS Y TABLA DE CONTENIDOS - OBLIGATORIO

### ⚠️ CRÍTICO: USAR SIEMPRE HeadingLevel PARA TÍTULOS Y SUBTÍTULOS

**NUNCA simules títulos con formato manual (negrita + tamaño grande).**
**SIEMPRE usa los Heading Levels nativos de Word.**

**¿Por qué es obligatorio?**
- Permite generar **Tabla de Contenidos automática** (TOC)
- Word reconoce la **estructura jerárquica** del documento
- Habilita navegación rápida en el panel de Word
- Es el estándar profesional y académico

### 📐 Definición de estilos de Heading:

```javascript
const doc = new Document({
  styles: {
    default: {
      document: {
        run: { font: "Times New Roman", size: 24 } // 12pt default
      }
    },
    paragraphStyles: [
      {
        id: "Heading1",
        name: "Heading 1",
        basedOn: "Normal",
        next: "Normal",
        quickFormat: true,
        run: { size: 32, bold: true, font: "Times New Roman" }, // 16pt
        paragraph: {
          spacing: { before: 240, after: 240 },
          alignment: AlignmentType.CENTER,
          outlineLevel: 0  // ⚠️ OBLIGATORIO para TOC
        }
      },
      {
        id: "Heading2",
        name: "Heading 2",
        basedOn: "Normal",
        next: "Normal",
        quickFormat: true,
        run: { size: 28, bold: true, font: "Times New Roman" }, // 14pt
        paragraph: {
          spacing: { before: 180, after: 180 },
          outlineLevel: 1  // ⚠️ OBLIGATORIO para TOC
        }
      },
      {
        id: "Heading3",
        name: "Heading 3",
        basedOn: "Normal",
        next: "Normal",
        quickFormat: true,
        run: { size: 26, bold: true, font: "Times New Roman" }, // 13pt
        paragraph: {
          spacing: { before: 120, after: 120 },
          outlineLevel: 2
        }
      }
    ]
  },
  sections: [{ children: [/* contenido */] }]
});
```

### 📋 Uso correcto vs incorrecto:

```javascript
// ✅ CORRECTO - Word lo reconoce como heading → permite TOC
new Paragraph({
  heading: HeadingLevel.HEADING_1,
  children: [new TextRun("INTRODUCCIÓN")]
}),

new Paragraph({
  heading: HeadingLevel.HEADING_2,
  children: [new TextRun("1.1 Contexto histórico")]
}),

// ❌ INCORRECTO - Word NO lo reconoce como heading → NO permite TOC
new Paragraph({
  alignment: AlignmentType.CENTER,
  children: [new TextRun({ text: "INTRODUCCIÓN", bold: true, size: 32 })]
}),
```

### 📑 Tabla de Contenidos automática:

```javascript
// Insertar TOC después de la carátula
new TableOfContents("Tabla de Contenidos", {
  hyperlink: true,
  headingStyleRange: "1-3"  // Incluye Heading 1, 2 y 3
}),
new Paragraph({ children: [new PageBreak()] }), // Salto después de TOC
```

**⚠️ REGLA CRÍTICA:** La TOC solo funciona si los títulos usan `HeadingLevel`. NO funciona con estilos manuales personalizados.

### 🎯 Cuándo incluir TOC:

| Tipo de documento | ¿Incluir TOC? |
|---|---|
| Ensayo corto (2-3 páginas) | No |
| Ensayo largo (5+ páginas) | Sí |
| Tarea/Cuestionario | No (salvo que tenga muchas secciones) |
| Informe técnico | Sí |
| Monografía | Sí, obligatorio |
| Respuesta argumentativa | No |

## 7. FORMATO DOCX-JS - REGLAS TÉCNICAS CRÍTICAS

### 📄 Configuración de página:

```javascript
sections: [{
  properties: {
    page: {
      size: {
        width: 12240,   // 8.5 pulgadas (US Letter) en DXA
        height: 15840   // 11 pulgadas en DXA
      },
      margin: {
        top: 1440,      // 1 pulgada = 1440 DXA
        right: 1440,
        bottom: 1440,
        left: 1440
      }
    }
  },
  children: [/* contenido */]
}]
```

**Tamaños de página en DXA (1440 DXA = 1 pulgada):**

| Papel | Ancho | Alto | Ancho contenido (márgenes 1") |
|---|---|---|---|
| US Letter | 12,240 | 15,840 | 9,360 |
| A4 (default) | 11,906 | 16,838 | 9,026 |

**Valores de interlineado (spacing.line):**

| Interlineado | Valor |
|---|---|
| Sencillo (1.0) | 240 |
| 1.15 | 276 |
| 1.5 | 360 |
| Doble (2.0) | 480 |

### 📝 14 REGLAS CRÍTICAS DE DOCX-JS (MEMORIZARLAS):

1. **Configurar tamaño de página SIEMPRE** - docx-js usa A4 por defecto
2. **NUNCA usar `\n`** - usar objetos `Paragraph` separados
3. **NUNCA usar bullets unicode** (`•`, `\u2022`) - usar `LevelFormat.BULLET` con config de numbering
4. **PageBreak DENTRO de Paragraph** - nunca suelto
5. **ImageRun requiere `type`** - siempre especificar png/jpg/etc
6. **Tablas: SIEMPRE `WidthType.DXA`** - NUNCA `WidthType.PERCENTAGE` (rompe Google Docs)
7. **Tablas necesitan doble ancho** - `columnWidths` en tabla Y `width` en cada celda
8. **Ancho tabla = suma de columnWidths** - deben cuadrar exactamente
9. **Usar `ShadingType.CLEAR`** - NUNCA `SOLID` para sombreado de tablas
10. **NUNCA tablas como divisores** - usar `border` en Paragraph
11. **TOC requiere HeadingLevel** - no estilos personalizados
12. **Incluir `outlineLevel`** en estilos de heading - obligatorio para TOC (0=H1, 1=H2, 2=H3)
13. **Siempre margins en celdas** - `{ top: 80, bottom: 80, left: 120, right: 120 }`
14. **Validar después de crear** - `python scripts/office/validate.py`

### 📊 Tablas correctas:

```javascript

const border = { style: BorderStyle.SINGLE, size: 1, color: "CCCCCC" };

new Table({
  width: { size: 9360, type: WidthType.DXA },        // Ancho total
```

## 8. CITAS Y REFERENCIAS APA 7 - COMPLETO

### ⚠️ CRÍTICO: APA 7 REQUIERE DOS COMPONENTES OBLIGATORIOS

1. **Citas en texto** (dentro del cuerpo del documento)
2. **Lista de referencias** (al final del documento)

**Ambos son obligatorios y deben ser coherentes entre sí.**

---

### 📝 CITAS EN TEXTO (en el cuerpo del documento):

#### Cita parentética (al final de la oración):
```
El compliance corporativo reduce significativamente los riesgos 
penales empresariales (Tiedemann, 2015).
```

#### Cita narrativa (el autor es parte de la oración):
```
Según Tiedemann (2015), el compliance corporativo reduce 
significativamente los riesgos penales empresariales.
```

#### Reglas según número de autores:

| Autores | Primera cita | Citas subsiguientes |
|---|---|---|
| 1 autor | (García, 2020) | (García, 2020) |
| 2 autores | (García y López, 2020) | (García y López, 2020) |
| 3+ autores | (García et al., 2020) | (García et al., 2020) |

#### Cita directa textual corta (menos de 40 palabras):
```
Tiedemann (2015) afirma que "el derecho penal económico requiere 
categorías propias" (p. 45).
```

#### Cita directa textual larga (40+ palabras) - en bloque indentado:
```
Como señala Silva Sánchez (2018):
    La expansión del derecho penal hacia ámbitos económicos no es 
    simplemente una extensión cuantitativa, sino que implica una 
    transformación cualitativa de las categorías dogmáticas 
    tradicionales. (p. 112)
```

#### Citas de normativa:
```
Primera vez:    Código Orgánico General de Procesos (COGEP, 2015, art. 22)
Subsiguientes:  COGEP, 2015, art. 22
```

#### Citas de jurisprudencia:
```
(Corte Constitucional del Ecuador, Sentencia No. 001-18-SEP-CC, 2018)
```

---

### 📚 LISTA DE REFERENCIAS (al final del documento):

**Formato APA 7 estándar:**

```
Libros:
Apellido, N. (Año). Título del libro en cursiva. Editorial.

Artículos de revista:
Apellido, N. (Año). Título del artículo. Nombre de la Revista 
    en cursiva, volumen(número), páginas. https://doi.org/xxxxx

Leyes/Códigos:
Asamblea Nacional del Ecuador. (2015). Código Orgánico General 
    de Procesos. Registro Oficial Suplemento N° 506.

Jurisprudencia:
Corte Constitucional del Ecuador. (2018). Sentencia 
    No. 001-18-SEP-CC.

Sitios web:
Autor/Entidad. (Año, día de mes). Título del artículo en cursiva. 
    Nombre del sitio. https://url

Capítulos de libro:
Apellido, N. (Año). Título del capítulo. En N. Editor (Ed.), 
    Título del libro en cursiva (pp. xx-xx). Editorial.
```

### 📖 Elementos mínimos:

**Para trabajos académicos:**
- Mínimo 5-8 referencias
- Incluir normativa citada (leyes, códigos)
- Incluir doctrina especializada (libros, artículos)
- Incluir fuentes web si se usaron
- Incluir jurisprudencia si aplica

### ⚠️ REGLAS DE COHERENCIA APA 7:

1. **TODO lo citado en texto debe aparecer en referencias** (y viceversa)
2. **Cursiva en títulos** de libros y revistas en la lista de referencias
3. **Sangría francesa** (primera línea normal, siguientes con sangría de 1.27 cm)
4. **Orden alfabético** por apellido del primer autor
5. **Si el mismo autor tiene varias obras del mismo año**: usar a, b, c → (García, 2020a), (García, 2020b)

### 💻 Implementación en docx-js:

## 11. PATRONES ANTI-IA - OBLIGATORIOS Y CRÍTICOS

### 🤖 ¿QUÉ SON LOS PATRONES ANTI-IA?

Son técnicas de escritura que hacen que el texto parezca **genuinamente humano** y no generado por IA. Los detectores de IA buscan patrones repetitivos, estructuras simétricas y falta de variación.

---

### ✍️ PATRÓN 1: VARIACIÓN EN LONGITUD DE ORACIONES

**❌ MAL (IA detectado - todas las oraciones similares):**
```
La dogmática penal tradicional se diseñó para delitos individuales. 
Los delitos económicos presentan características diferentes. 
La responsabilidad penal debe adaptarse a nuevas realidades. 
El compliance corporativo es una herramienta fundamental.
```

**✅ BIEN (Humano - variación natural):**
```
La dogmática penal tradicional se diseñó para delitos individuales. 
Pero cuando una corporación defrauda al fisco mediante estructuras 
societarias complejas que involucran múltiples jurisdicciones y 
operaciones coordinadas entre diversos ejecutivos, ¿cómo aplicamos 
conceptos pensados para el ladrón que roba un celular? El compliance 
corporativo surge como respuesta.
```

**MEZCLA OBLIGATORIA:**
- **Cortas** (5-12 palabras): Para impacto, transiciones
- **Medianas** (15-25 palabras): Para desarrollo normal
- **Largas** (30-45 palabras): Para explicaciones complejas

---

### ✍️ PATRÓN 2: EVITAR ESTRUCTURAS SIMÉTRICAS

**❌ MAL (IA detectado - simetría robótica):**
```
En primer lugar, la dogmática tradicional no contempla bienes 
jurídicos colectivos. En segundo lugar, los delitos económicos 
requieren análisis transnacional. En tercer lugar, la identificación 
de autores es compleja. En cuarto lugar, el compliance es necesario.
```

**✅ BIEN (Humano - estructura variada):**
```
La dogmática tradicional ignora un problema fundamental: los bienes 
jurídicos colectivos. ¿Cómo proteger "el orden económico" cuando 
no hay una víctima identificable? Además, estos delitos cruzan 
fronteras. Un fraude corporativo puede ejecutarse en Ecuador, 
afectar inversionistas en Estados Unidos y ocultar activos en 
paraísos fiscales. Identificar al autor se vuelve casi imposible 
cuando participan cincuenta ejecutivos en una cadena de decisiones. 
Aquí es donde el compliance cobra sentido.
```

**TÉCNICAS PARA VARIAR:**
- Alterna afirmaciones directas con preguntas
- Combina párrafos largos con párrafos cortos
- Usa ejemplos concretos entre explicaciones teóricas
- Intercala datos con reflexiones

---

### ✍️ PATRÓN 3: CONECTORES NATURALES Y VARIADOS

**❌ MAL (IA detectado - conectores repetitivos):**
```
Además, el delito económico presenta complejidad.
Además, la autoría es difusa.
Además, requiere especialización.
Por otro lado, la dogmática tradicional es insuficiente.
Por otro lado, necesitamos nuevas categorías.
```

**✅ BIEN (Humano - conectores variados):**
```
Pensemos en un caso real: una empresa defrauda al fisco. 
La complejidad es evidente. Ahora bien, ¿quién es el autor? 
El CFO autorizó, pero el gerente general decidió, mientras 
que el contador ejecutó. Veamos otro problema: la dogmática 
clásica no tiene respuestas. Aquí surge una pregunta inevitable: 
¿necesitamos categorías nuevas o reinterpretar las existentes?
```

**USA VARIEDAD:**
- Transiciones naturales: "Pensemos en...", "Veamos ahora...", "Consideremos..."
- Preguntas: "¿Qué implica esto?", "¿Cómo resolver...?"
- Contrastes: "Sin embargo,", "No obstante,", "Por el contrario,"
- Causales: "Esto se debe a...", "La razón radica en...", "Dado que..."
- Ejemplos: "Por ejemplo,", "Tomemos el caso de...", "Ilustremos con..."

---

### ✍️ PATRÓN 4: PREGUNTAS RETÓRICAS

**❌ MAL (IA detectado - solo afirmaciones):**
```
La inteligencia artificial plantea desafíos al derecho penal.
No podemos imputar responsabilidad a una máquina.
El programador no siempre es responsable.
La empresa debe responder en algunos casos.
```

**✅ BIEN (Humano - preguntas que generan engagement):**
```
La inteligencia artificial plantea desafíos al derecho penal. 
Pero, ¿cómo imputar responsabilidad a una máquina? Si un 
algoritmo comete fraude, ¿culpamos al programador que lo 
diseñó hace cinco años? ¿O a la empresa que lo desplegó sin 
controles? ¿Acaso la IA misma puede ser sujeto de 
responsabilidad penal?
```

**CUÁNDO USAR PREGUNTAS:**
- Para introducir problemas complejos
- Para generar transiciones entre ideas
- Para involucrar al lector en el razonamiento
- Para plantear dilemas sin respuesta fácil

---
